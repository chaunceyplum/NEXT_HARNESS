# Scope-gated build planning: stop synthetic defaults from cascading into Launch/CJA/AJO (v2)

This pass reviews a single follow-up commit (`3b75702`) addressing v1's Issue #2: `cjaModule.build()` and `ajoOffersModule.build()` still fell back to the literal string `'engaged_visitors'` for a segment name whenever `config.segments` was empty, even though the segment that name implied had never been created anywhere in the plan since the earlier commits removed `aepModule`'s synthetic default. The fix replaces the CJA fallback with a vertical-derived standalone segment name (`<vertical>_reporting_segment_cja_<token>`) and has `ajoOffersModule` omit the `eligibility_rules.segments` key entirely rather than reference a phantom segment. Two new regression fixtures (G, H) reproduce the exact scenarios cited in the v1 review and assert the literal string `'engaged_visitors'` no longer appears in the resulting plan.

Watch for: this commit does not touch `lib/intent-extractor.ts` (v1 Issue #1) or add any live-model verification for the LLM synthesis prompt (v1 Issue #3) — both were explicitly scoped out by the orchestrator as separate/out-of-environment concerns, not overlooked (**confirmed**, per orchestrator briefing).

## High-level view

The fallback removal is a straight substitution: `segments[0]?.name || 'engaged_visitors'` becomes `segments[0]?.name` (now genuinely `undefined` when there's no real segment), and both call sites branch on that instead of silently naming things after a segment that doesn't exist. CJA gets a real standalone baseline segment with honest naming and description; AJO offers drops the eligibility gate entirely rather than gating on nothing. Confirmed directly via `Fixture G` (reporting-goal-only config produces `cja_segment` named `retail_reporting_segment_cja_<token>`, no `engaged_visitors` anywhere) and `Fixture H` (web-destination-only config produces an `ajo_offer_0` step with `eligibility_rules: {}`).

The two new fixtures reproduce the exact configs the v1 review used to demonstrate the bug rather than a generic smoke test, making them an effective guard against the same regression recurring. All 8 fixtures (A–H) pass, `npm run build` exits 0, and `npm run lint` shows the same 54 pre-existing baseline errors as before.

The two remaining v1 issues (intent-extractor keyword cascades, unverified live-model prompt guard) are unchanged by this commit and remain open per the orchestrator's explicit scoping decision — Issue #1 is a distinct mechanism from what this task targeted, and Issue #3 is an environment limitation rather than a code defect.

### The orphaned segment reference is gone, cleanly

`cjaModule.build()`'s `primarySegmentName` is now typed `string | undefined` and both the step label and the segment name/description branch on its presence — a real segment gets mirrored with the old naming scheme (`${primarySegmentName}_cja_${ctx.runToken}`), and its absence gets an honestly-labeled baseline segment (`Create a baseline CJA reporting segment`) named from the vertical via the existing `slug()` helper rather than from a segment that was never created. `ajoOffersModule.build()` takes the simpler route of omitting `eligibility_rules.segments` entirely rather than inventing a replacement name — appropriate here since an offer with no eligibility segment is a legitimate (if broader) config, unlike CJA where a segment step is expected to exist.
