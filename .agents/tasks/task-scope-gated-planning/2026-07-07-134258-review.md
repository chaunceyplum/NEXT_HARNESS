# Scope-gated build planning: stop synthetic defaults from cascading into Launch/CJA/AJO (v1)

This change fixes the deterministic plan builder (`lib/plan-builder.ts`) so a narrow request like "build me a schema in AEP" produces only the AEP (and RAG) steps it actually needs, instead of also standing up Adobe Launch instrumentation, CJA reporting, and AJO activation. The root cause was three "empty means yes" defaults: `aepModule` synthesized a fake `engaged_visitors` segment whenever none was requested, `classifyUseCase()` treated an empty `destinations` array as an analytics signal, and `ajoActivationModule` treated the same empty array as "activation assumed by default." All three were removed, and the now-dead default-fallback code in `launchModule`/`ajoActivationModule`/`cjaModule`'s `build()` was cleaned up to match. A parallel change adds a "STAY IN SCOPE" instruction with a concrete negative example to the LLM synthesis prompt for when `ANTHROPIC_API_KEY` is configured. A new verification script, `scripts/verify-planning.ts`, exercises the fix directly since the repo has no test framework.

Watch for: the fix doesn't touch `lib/intent-extractor.ts`'s regex enrichment, which can still independently pull `cja`/`ajo_offers` into requests that read as narrow to a human (**confirmed**). A leftover `'engaged_visitors'` fallback in `cjaModule.build()`/`ajoOffersModule.build()` fires for CJA-included plans triggered purely by a reporting goal with zero real segments, referencing a segment that was never created (**confirmed**). The LLM-prompt scope guard has no code-level enforcement backstop and is unverified against a live model since no API key is configured here (**confirmed**, acknowledged in the task's own findings).

## High-level view

The core fix removes three "absence of a signal treated as a synthetic default" branches and lets the removal of one (the fake AEP segment) organically starve the two downstream modules that keyed off "a segment exists." Fixing the upstream synthesis point rather than patching each downstream gate independently is the right call — it removes the false signal at its source instead of duplicating the "is this real" check three times.

The regex-based intent enrichment in `lib/intent-extractor.ts` is untouched by this diff, but it sits upstream of the exact modules this fix targets and re-adds segments/placements from broad keyword matches even on requests that read as narrowly scoped — "build a schema for our shopping cart" still pulls in an `engaged_browsers` segment and a `checkout_offer` placement via keyword match, cascading into `cja`/`ajo_offers` through a different path than the one this diff fixed.

The LLM synthesis prompt change adds a specific "STAY IN SCOPE" instruction with a worked negative example, and `validateSynthesizedSteps()` gained a comment explaining why it intentionally doesn't enforce category scope — it only sees synthesized steps, not the raw request. Enforcement here is entirely prompt-based with no code-level backstop if the model ignores the instruction; that's a documented tradeoff, not an oversight, but it's untested against a live model in this environment.

Test coverage is a purpose-built script rather than a real test suite, run manually since there's no framework in this repo. It exercises the three fixed gates plus the prompt text and a regression fixture for the legitimate multi-module case, but doesn't touch the intent-extractor path at all.

### Root-cause fix: removing the segment/destination "empty means yes" defaults

The three changes in `lib/plan-builder.ts` all follow the same shape: a branch that treated an empty array as a positive signal is deleted.

```
// classifyUseCase() — before
const analyticsSignals = goals.some(...) || destinations.length === 0;
// after
const analyticsSignals = goals.some(...);
```

```
// ajoActivationModule.isApplicable() — before
const wantsActivation = destinations.length === 0 || destinations.some(d => [...].includes(d));
// after
const wantsActivation = destinations.some(d => [...].includes(d));
```

```
// aepModule.build() — before
const segmentList = segments.length > 0 ? segments : [{ name: 'engaged_visitors', ... }];
// after
segments.forEach(...)   // segments only, no synthetic entry
```

The AEP change is the one that matters most structurally: removing the fake segment there is what starves `cjaModule` and `ajoActivationModule`'s `hasSegments` checks downstream, since those two never had their own synthetic-segment problem — they were reading a config that AEP had already polluted. `launchModule`'s dead `page_view` fallback and `ajoActivationModule`'s dead `engaged_visitors` fallback in `build()` were removed as unreachable code (their `isApplicable()` gates already required non-empty arrays), with explanatory comments left in place rather than defensive fallbacks — a silent fallback there would mask a future `isApplicable()` regression rather than surface it.

Verified directly: `planUseCase()` against a schema-only config (domain + vertical set, everything else empty) now includes only `rag` + `aep`, with zero `aep_segment_*` steps. A full-activation fixture (events + segments + `email` destination + a reporting goal) still includes `launch` + `cja` + `ajo_activation`.

### CJA/AJO-offers still reference a never-created `'engaged_visitors'` segment

`cjaModule.build()` and `ajoOffersModule.build()` still contain `segments[0]?.name || 'engaged_visitors'`. A config with a reporting goal (`profile.analyticsFocused === true`) but zero segments now correctly includes `cjaModule` (via `analyticsFocused`, not `hasSegments`), and `cjaModule.build()` runs with `segments = []`, producing a CJA segment named `engaged_visitors_cja_<token>` whose eligibility/definition references a segment that was never created in AEP. Verified directly: a config with `goals: ['improve_reporting']` and empty segments/events/destinations produces `modules: rag, cja` and a `cja_segment` step named `engaged_visitors_cja_<token>` with no corresponding AEP segment step anywhere in the plan. Same "empty means yes" pattern this diff otherwise removed, one level down in the same module.

### Intent-extractor enrichment isn't gated by this fix

`enrichConfigFromDescription()` re-scans the raw description with regex keyword tables and adds events/segments/placements/goals/destinations on top of whatever the upstream MCP planner returned, before `planUseCase()` ever runs. The file is untouched by the diff, and the task's findings note it was checked against the literal string "build me a schema in AEP" and found clean. Slightly less minimal phrasing false-positives: "build me a schema in AEP for our shopping cart" matches `SEGMENT_KEYWORDS`'s `/\b(cart|browse|window shop|considering)\b/i` (adding `engaged_browsers`) and `PLACEMENT_KEYWORDS`'s checkout regex (adding `checkout_offer`), flowing into `ajo_offers` and `cja` being included — the same cascade the user reported, triggered from a different code path than the one this diff fixed.

### LLM synthesis prompt: scope guard is prompt-only, unverified against a live model

`validateSynthesizedSteps()`'s new comment explains it can't enforce category scope because it never sees the raw request, only the synthesized steps — a fair architectural tradeoff, but it means there is no code-level backstop for this specific failure mode. If the model ignores the "STAY IN SCOPE" instruction, an over-scoped plan reaches the runner as long as it's otherwise well-formed (valid tools, params, refs). No `ANTHROPIC_API_KEY` is configured in this sandbox, so this half of the fix is verified only by confirming the instruction text renders into the prompt string, not by observing actual model behavior.

### Test coverage: purpose-built script, not a real test suite

`scripts/verify-planning.ts` covers the three fixed gates directly (schema-only, full-activation regression, analytics-only) plus the prompt-text check and the validator's documented non-enforcement behavior. It does not exercise `enrichConfigFromDescription()` at all, so the interaction between regex enrichment and the newly-tightened gates (the shopping-cart scenario above) has no automated coverage either way. It's a standalone script, not wired into any CI target, so it won't run automatically — someone has to remember to invoke it.
